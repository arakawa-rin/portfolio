LOGIC 01 精算ロジック（誰から誰へいくら払うかの計算）
処理の概要
各参加者の「立替合計（total_paid）」と「負担合計（total_allocations）」をDBから集計し、その差額（net）をもとに精算ペアと金額を自動計算する処理です。
net がプラスの人（受け取り側）とマイナスの人（支払い側）に分けてリスト化し、貪欲法で順番に照合しながら精算の組み合わせを最小化しています。
ソースコード
# 各参加者の立替合計・負担合計をDBから集計
allocations_dict = {row.ep_id: row.total_allocations for row in allocations_row}
expenses_dict    = {row.ep_id: row.total_paid       for row in expenses_row}

settlement_summary = []
for p in participants:
    ep_id      = p.event_participant_id
    total_alloc = allocations_dict.get(ep_id, Decimal('0'))
    total_paid  = expenses_dict.get(ep_id,    Decimal('0'))
    net = total_paid - total_alloc          # プラス = 受け取り権あり
    settlement_summary.append({
        "event_participant_id": ep_id,
        "display_name":         p.display_name,
        "total_allocations":    total_alloc,
        "total_paid":           total_paid,
        "net":                  net,
    })

# 整合性チェック（貸借の合計が 0 でなければエラー）
total_balance = sum((b["net"] for b in balances), Decimal('0'))
if total_balance != Decimal('0'):
    return render_template('settlements.html', event=event,
                           error="内部エラー: 貸借の合計が0になっていません")

# 受け取り側・支払い側に分けてソート
creditors = sorted([b for b in balances if b["net"] > 0],
                   key=lambda x: x["net"], reverse=True)
debtors   = sorted([b for b in balances if b["net"] < 0],
                   key=lambda x: x["net"])

# 貪欲法で精算ペアを生成
settlements = []
while creditors and debtors:
    creditor = creditors[0]
    debtor   = debtors[0]
    settlement_amount = min(creditor["net"], -debtor["net"])
    settlements.append({
        "payee_display_name": creditor["display_name"],
        "payer_display_name": debtor["display_name"],
        "amount":             settlement_amount
    })
    creditor["net"] -= settlement_amount
    debtor["net"]   += settlement_amount
    if creditor["net"] == Decimal('0'): creditors.pop(0)
    if debtor["net"]   == Decimal('0'): debtors.pop(0)
工夫した点 / 意識した点

・立替合計と負担合計の差額（net）を計算した後、貸借の合計が必ず 0 になるという性質を整合性チェックとして活用しています。合計が 0 でない場合は内部エラーとして画面に通知する設計にしました。

・精算ペアの生成には貪欲法を採用し、受け取り額の大きい人・支払い額の大きい人から順に照合することで、精算回数を最小化しています。

・金額計算には Python の Decimal 型を使用し、浮動小数点誤差によるバグを防いでいます。

LOGIC 02 支出登録ロジック（バリデーション・カテゴリ自動登録・割り勘配分）
処理の概要
支出の登録リクエストを受け取り、入力値のバリデーション・カテゴリの自動登録・支出レコードの挿入・割り勘配分計算を順番に処理するAPIルートです。
複数の処理を1リクエスト内でトランザクション管理し、途中でエラーが発生した場合はロールバックして整合性を保ちます。
ソースコード
deta = request.get_json()
category_name  = deta.get('category_name')
expense_amount = deta.get('expense_amount')
payer_ep_id    = deta.get('payer_ep_id')
participants_ids = deta.get('participants')
expense_at     = deta.get('expense_at')
split_method   = deta.get('split_method')

# 必須項目チェック
if not category_name or not expense_amount or not payer_ep_id \
        or not participants_ids or not expense_at:
    return jsonify({"success": False, "error": "入力してください"}), 400

# 日付・金額のバリデーション
try:
    expense_at = datetime.strptime(expense_at, '%Y-%m-%d').date()
except ValueError:
    return jsonify({"success": False, "error": "日付が不正です"}), 400

try:
    expense_amount = Decimal(expense_amount)
except InvalidOperation:
    return jsonify({"success": False, "error": "金額が不正です"}), 400

if expense_amount <= 0:
    return jsonify({"success": False, "error": "金額が不正です"}), 400

# イベント期間外の支出を弾く
if event.start_date > expense_at or expense_at > event.end_date:
    return jsonify({"success": False, "error": "日付が不正です"}), 400

# 立替者がイベント参加者かチェック
if not Event_Participants.query.filter_by(
        event_id=event_id, event_participant_id=payer_ep_id).first():
    return jsonify({"success": False,
                    "error": "立替人がイベント参加者ではありません"}), 400

# カテゴリが未登録なら自動で登録（重複を防ぐ）
category = Expense_Categories.query.filter_by(
    event_id=event_id, category_name=category_name).one_or_none()
if category is None:
    category = Expense_Categories(category_name=category_name, event_id=event_id)
    db.session.add(category)
    db.session.flush()

# 支出レコード挿入
expenses = Expenses(event_id=event_id, expense_amount=expense_amount,
                    payer_ep_id=payer_ep_id, expense_at=expense_at,
                    description=description, category_id=category.category_id)
db.session.add(expenses)
db.session.flush()

# 割り勘配分の計算・挿入
alloc_error = create_allocations(
    expenses.expense_id, event_id, expense_amount,
    split_method, participants_ids, deta)
if alloc_error:
    return jsonify({"success": False, "error": alloc_error}), 400

db.session.commit()
return jsonify({"success": True, "expense": expenses.to_dict()})
工夫した点 / 意識した点

・バリデーションを「必須チェック → 型変換 → 業務ルールチェック」の順に段階的に行い、エラーの原因が特定しやすいよう個別にメッセージを返しています。

・カテゴリは同名のものが既に存在する場合は再利用し、なければ自動登録する設計にしました。ユーザーが毎回カテゴリを選ぶ手間を省きつつ、重複を防いでいます。

・flush() で支出IDを確定させてから割り勘配分計算に渡すことで、コミット前でも関連レコードの挿入が正しく行えるようにしています。

LOGIC 03 イベント作成ロジック（ゲスト対応・招待コード生成・参加者一括登録）
処理の概要
イベントの作成リクエストを受け取り、ゲストユーザーの自動採番・イベントレコードの挿入・招待コードの生成・参加者の一括登録をまとめて処理するAPIルートです。
アカウント未登録のユーザーでもイベントを作成できるよう、セッションにユーザーIDがなければゲストユーザーを自動生成します。
ソースコード
deta = request.get_json()
event_name   = deta.get('event_name')
start_date   = deta.get('start_date')
end_date     = deta.get('end_date')
display_name = deta.get('display_name')

# 必須項目・日付バリデーション
if not event_name or not start_date or not end_date or not display_name:
    return jsonify({"success": False, "error": "入力してください"}), 400
try:
    start_date = datetime.strptime(start_date, '%Y-%m-%d').date()
    end_date   = datetime.strptime(end_date,   '%Y-%m-%d').date()
except ValueError:
    return jsonify({"success": False, "error": "日付が不正です"}), 400
if start_date > end_date:
    return jsonify({"success": False, "error": "日付が不正です"}), 400

# セッションにユーザーIDがなければゲストユーザーを自動生成
user1 = session.get("user_id")
if user1 is None:
    user1 = guest_create()

# イベント登録
events = Events(event_name=event_name, start_date=start_date,
                end_date=end_date, created_by=user1)
db.session.add(events)
db.session.flush()      # event_id を確定させる

# 作成者を参加者として登録
event_participants = Event_Participants(
    event_id=events.event_id, user_id=user1,
    display_name=display_name, status='joined')
db.session.add(event_participants)

# 追加メンバーを一括登録（ゲスト扱い：user_id = None）
member_names = deta.get('member_names', [])
for name in member_names:
    name = name.strip()
    if name:
        member_participant = Event_Participants(
            event_id=events.event_id, user_id=None,
            display_name=name, status='joined')
        db.session.add(member_participant)

# 重複しない招待コード（6桁英数字）を生成
while True:
    code = ''.join(random.choices(string.ascii_uppercase + string.digits, k=6))
    if not Events.query.filter_by(invite_code=code).first():
        events.invite_code = code
        break

db.session.commit()
return jsonify({"success": True, "event_id": events.event_id})
工夫した点 / 意識した点

・アカウント不要でイベントを作れるよう、セッションにユーザーIDがない場合はゲストユーザーを自動採番する設計にしました。追加メンバーも user_id = None で登録し、アカウントなしで参加者として扱えます。

・招待コードはDBに存在しない6桁英数字が生成されるまでループする方式で、コードの一意性をアプリ層で保証しています。

・flush() でイベントIDを確定させてから参加者レコードを挿入することで、コミット前でも外部キー制約が正しく機能するようにしています。
